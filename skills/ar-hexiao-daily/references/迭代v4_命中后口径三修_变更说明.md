# 迭代 v4 变更说明（2026-07-24）：命中之后的三条口径修复

> 触发：2026-07-23 明昊又去明妹工位实测（v1.0.20，opencode 免费模型跑桌面真实 13 笔）。
> 主体已对（143 命中 / 14 挂起 / 0 冲突 / 0 异常，她当场认可"分析都是对的"）。
> 这一版只修她**逐笔口述**出来的、命中之后"怎么填/怎么判"的三个缺口——**不动取数、不动挂起判断**。
>
> 一句话：**取数与判定都没错，错在"填进去那一下"的三条口径。**

## 一、修了什么（都在 record 命中盈亏表**之后**）

| # | 问题（她原话/实测） | 老行为 | 新行为 |
|---|---|---|---|
| **1** | **交付额中途变了**要以智云为准（口述 SO107/422：初始交付 → 结算变了）| 靠 SOD/SO 唯一行命中的那几行，**闷声按智云额填、一声不吭**（实测 SO26040322 有 2 行这样静默）| 命中行若**智云额 ≠ 她表应收**，一律顶 ⚠ 到「怎么办」："已按智云额填、扫一眼"。超额 E4 上限也从"她表应收"改成"智云交付额" |
| **2** | 流转表「是否更新」把**部分核销**误判成「部分」（实测 AR26070112）| 一笔到账里只要有一个挂起（含 E5 部分核销）就掉「部分」| 只有**没交付进表**(E2/E3) 才让它掉「部分」；E5（钱已全核、她拆行）算**更新了** → 「是」|
| **3** | 没回满该**拆成两行**、她说程序"没学会"（口述 AR26070105）| E5 只说一句"插一行拆已收/未收"，没数字 | 把**两行的具体金额算给她**：已收行=实到额/结账「是」/计提留空；未收行=差额/结账「否」|

## 二、为什么这么改（口径出处）

- **①交付额变化**：明妹原话"要让他动脑子去检查交付额，别以我表里那个为准。上面交付额跟下面核销额
  一样就以它为准，按实际核销金额回填"。→ 写进去的一律是智云额（本就如此），**新增的是"每一行都
  被明确检查一次、对不上就 ⚠"**。老版只有「整段等比例差(×0.97)」路径会提醒，靠 SOD/SO 命中的漏了。
  另：超额 E4 拿"她表应收"当上限是同一个病——交付额变**大**时（初始 1 → 结算 2），旧应收当上限会把
  "本该填的 2"误报超额。改用智云交付额当上限。
- **②流转部分**：明妹原话"部分是指某些订单没交付、我没法更新才叫部分"。E5 部分核销的钱**已经全部
  核销落地**，她只是把一个订单拆成已收/未收两行——这算更新。实测 AR26070112：12 单里 1 单 E5 →
  旧版整笔「部分」、新版「是」。
- **③拆行**：明妹口述那笔（AR26070105：该收 8946.89 / 实到 8729.35 / 差 217.54）"回了 8729 算回完、
  结账变是，剩下 217 新开一行变否"。程序把这两行数字算给她照做。**不代拆**——插行会动她表里的
  透视/公式/图形对象引用，风险高，与"只改命中行几个格、其余字节不动"的写入铁律一致。

## 五、没动的（别以为改了）

- 取数（单入口 + 关联子表 + SOD 级）——v3 架构原样。
- 分笔 hold / 手续费 hold / 尾差阈值 0 / 跨年老单 hold ——口径未拍板，原样。
- 永不写智云；只碰盈亏明细副本；流转表只建议不自动写。
- **拆行仍由她手动执行**——程序只把方案算给她，不代插行。
