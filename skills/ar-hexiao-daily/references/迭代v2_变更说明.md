## 修了什么

| # | 缺口 | 怎么修 |
|---|---|---|
| 1 | **到账流转表根本没接进判定**：`flow_hits` 只读不写，R2 三键匹配未实现，E0/E12 是死码 | 新增 `scripts/flow_ledger.py`：**按表头内容认表**（不看文件名，她各渠道分好几张）；匹配键按可靠度排序 —— ①日期+净额+公司名 ②日期+(净额+手续费)+公司名（微信/支付宝把客户支付额拆成净额与手续费）③日期+金额但名字不符=**弱命中**须人工确认。清单新增「这笔到账在流转表哪一行」「流转表单号列建议填」 |
| 2 | **rescan 只合并不重判**：上月挂起的笔不会再出现在当日导出里 → 永远捞不回来，第 8 步价值为零 | `rescan_holds.py` 新增 `reclassify_against_ledger()`：**拿当前盈亏表对挂起项重新判定**。本地判得动的（E2 未交付 / E3 跨年 / E8 多行）自动升级「可补做」；E1/E4/E9 等必须新导出才知道的**如实保留**并说明 |

| 4 | **无 per-AR 核销合计校验**：E4 只按单条判，同一 AR 多行合计超过到账查不出来 | `check_ar_totals()`：Σ本次核销 ≤ 到账（净额/含手续费两种口径取宽松者）；超出 → 该 AR 全部行 E4 异常。拿不到到账总额就**不判**（不猜） |
| 5 | 混合案例表达不了：同一到账下部分 SO 能填、部分不能 | 新增 `build_ar_summary()` + 清单第 4 页签「按到账汇总」，派生流转表「是否更新应收款」**三态：是 / 部分 / 空白**（这正是她表里出现"部分"的原因，实测微信表 2 行、汇款表 1 行） |
| 6 | 只在文档里承诺"不改你的表"，没有机器可验的证据 | 新增 `scripts/verify_sources.py`（**思路借鉴李尚 `verify_source_hashes.py`**）：跑前 `snapshot` 记 sha256，跑后 `verify` 比对，任何改动/丢失/多出文件 → 非 0 退出 |
| 7 | 挂账没写"等什么才能回来做" | 台账新增「复查条件」列，按 E 码给出可执行条件（E2→月初贴完交付；E3→拿到对应年度盈亏表…） |
| 8 | 部分核销没提醒加行 | 清单「今天能填」新增「部分核销提醒」列（**只提醒，不替她插行**——与发她的方案一致） |

## 端到端实跑（真实数据）

夹具 53 笔 + 真实全年盈亏表 + 两张真实流转表（640 行）：

- 判定：auto 27 / hold 26（E3 跨年 25、E2 未交付 1）/ exception 0
- 重扫：23 个案例，**1 笔经重判升级为「可补做」**（证明第 8 步真的活了）；连跑两遍计数一致（幂等）
- **源文件只读校验通过**：3 个副本 sha256 与运行前完全一致

## 实跑抓到、文档没写过的两个坑

1. **盈亏表「明细」会被误认成流转表**：它也有"单号 / 客户名称 / 项目下单日期 / 应收金额"，宽松匹配把 5000 行明细当成了流转行。
   → 认表加硬条件：必须有流转表专有列（是否更新应收款 / 是否已登记系统 / 收款形式），且出现盈亏专有列（计提金额 / 新智云单号…）即否决。
2. **只给一两张渠道表就判 E0 会大面积冤枉**：实测 53 笔误报 28 笔"对不到账"。
   → **默认不判 E0**；只有显式 `--flow-complete`（声明当天所有渠道流转表都给全了）才判。

## 仍未做（不许对外说已做）

- **预存通道拿不到 SO**：智云明细子表接口不返回订单号，须与「下单栏」按金额配对。
  **根因是数据契约漏了「下单栏」这份输入**——要先补契约再改码。这类目前全部 hold。
- 预收余额链（`上次余额 - 本次核销 = 本次余额`）未建模，需历史年度流转表。
- 手续费口径、尾差阈值、一 SO 多行落点、跨年策略 —— 四项待明妹拍板，一律挂起。

## 吸收自李尚实现的点

- 匹配键把「净额 + 手续费」作为**匹配**口径（绕开尚未拍板的**回填**口径），
- `case_id` 作为跨轮次稳定主键 + 每个案例写「复查条件」，
- 流转表「是否更新应收款」三态模型（是/部分/空白）与"必须保存每个 SO 的结构化状态"，
- 源文件哈希校验作为只读保证，
- 「不默认忽略小额尾差：录屏只证明个案，推不出通用阈值」这一条更准的证据解读。

## opencode 真实模拟测试（2026-07-22 · 免费模型 deepseek-v4-flash-free）

按财务部skills标准规程③做的"真实模拟测试"——装进本机 opencode，用明妹的原话跑真实数据。
**结论：脚本层没问题，AI 编排层第一轮不可靠，已修三处后通过。**

### 第 1 轮（失败）：说"跑昨天的核销"，它从第 2 步挑收入开始

SKILL.md 的 11 步表被当成"从头顺着走"，跑完 extract_income 就停在第 3 步等她贴流转表——
而她要的是第 6 步判定。**日更工具这样没法用。**

### 第 2 轮（更糟）：绕开脚本自己写代码 dump 表格

免费模型直接写了个 Python 把银行日记账各 sheet 原样打印进对话，
**真实客户名和金额进了聊天记录 → 直接违反红线 2**。之后判定脚本一次也没调，
还把"银行日记账"说成"回款记录"让她去导出。

### 修了三处

1. **SKILL.md 顶部加「意图 → 步骤」路由表**：她说什么直接跳哪一步，明写"跑昨天的核销 = 第 6 步，不是第 2 步"。
2. **加「硬禁止」小节**：禁止 AI 自己写脚本读她的 Excel（只准调 `scripts/`）、禁止把表格内容回显对话、缺输入就停下报缺并给出照抄文案。
3. **`classify_hexiao` 兜底认 `01_智云导出/*.json` 取数夹具**，缺件时给明确文案而不是让 AI 自由发挥。

### 第 3 轮（通过）

路由正确 → `inspect_inputs` → `verify_sources snapshot` → `classify_hexiao`（53 笔：auto 27 / hold 26 / exc 0）
→ `build_worklist` → `verify_sources verify` 通过。**对话里没有出现任何客户名与金额明细**，只报计数与 E 码。

### 第 3 轮仍发现的一处危险措辞（已修）

它结尾对她说"把自动判定的 **27 笔填进智云**"——**错且危险**：该填的是她自己的**盈亏核算表**，
智云是我们永不碰的系统。AI 自由发挥交接话术就会出这种事。
→ SKILL.md 加**逐字照抄的结尾模板**，并写死 ⛔"绝不能说 填进智云 / 录入智云 / 同步到系统"。

### 教训（值得进坑点台账）

**"技能=AI 壳 + 确定性脚本"里，壳同样会翻车，而且翻的是对用户说的话。**
凡是会让她动手的交接句子，必须在 SKILL.md 里给逐字模板，不能让模型自己组织。

## v2.4 · 按官方 skill 最佳实践优化（2026-07-22）

先查了怎么做好 skill（本地 `skill-creator` 技能 + [官方 Skill authoring best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)），
对照 checklist 发现我们缺一个官方力推的模式，正好是"尽可能自动化"的安全实现路径：

### plan → validate → execute

官方原话：批量/破坏性/高风险操作要先产出**可校验的中间产物**，用脚本校验通过再执行。
我们的判定结果本来就是一份"打算怎么填"的计划，缺的是**写之前的复核**。

新增 `scripts/validate_plan.py`：拿判定结果对**当前**盈亏表逐条复核，分三种结论——

| 结论 | 条件 | 处理 |
|---|---|---|
| `write` | 行号仍指着同一 SO/SOD、五列为空、值合法 | 可以写 |
| `skip` | 已填过且与本次算的一致 | 幂等跳过（重复跑不重复写） |
| `conflict` | **行号对不上**（她插过行）/ 已填但不一致 / 值不合法 / 同一行被两笔命中 | 不写，写清原因交给人 |

**为什么必须有**：从判定到写入之间她的表可能已经变了（月初贴交付插行、部分核销上方插行 → **行号立刻失效**）。
按失效的行号写 = 把值写到别人家的行上。

### 这一步第一次跑就抓到一个真规则 bug

拿 7-08 那 53 笔跑校验：**26 笔幂等跳过、1 笔冲突**。冲突这笔 `AR26070033|SO26060469`：

- 表里她填的收款方式是 **冲预收**，我们算的是 **汇**
- 查明：这笔 **6-26 到账、7-08 核销（跨月）**
- 规则 R6 的「乙·晚核销标签」——钱早到账、销售下月才关联订单 → 收款方式写「冲预收」，**这条根本没实现**，`pay_way()` 只看核销状态和回款类型

修完重跑：**冲突归零，27 笔与她当时手工填的逐格一致（27/27）**。

> 这条标签是给月度提成筛选用的，填错就漏提成——**靠合成测试永远发现不了，只有拿她的历史真数据回放才照得出来**。
