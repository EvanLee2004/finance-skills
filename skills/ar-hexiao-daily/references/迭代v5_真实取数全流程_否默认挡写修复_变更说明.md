# 迭代 v5 变更说明（2026-07-24）：真实取数全流程实测 → 修「否默认挡写」bug

> 触发：2026-07-24 明昊要求**真·全流程实测**——不用罐头数据，让 opencode 从第 6 步「拿智云数」开始。
> 明妹智云账号（getpass 运行时输、密码不落盘，Claude 不经手）→ `fetch_zhiyun.py` 只读抓 **2026-07-24 核销**
> 4 笔真实到账 → opencode 跑判定链。**这一跑照出金标回放永远照不出的一个真 bug。**
>
> 一句话：**取数、判定全对；错在 validate 把她表预置的「是否结账=否」当成"她已经填过这一行"，
> 于是每一笔新到账都被误判成冲突、自动回填对真实新数据全线失效。**

## 一、bug 现场（2026-07-24 真实 24 号数据）

- `fetch_zhiyun.py` 真实抓到：4 笔核销日期=2026-07-24 到账（整笔 1 / 预存 1 / 空 2），10 SO / 10 SOD，
  `无下单行的AR: []`、`查不到SOD的SO: []`——取数干净。
- classify：可填 4 / 挂账 6（E2 未交付 1 / E3 跨年 3 / E5 部分 1 / **E_FEE 手续费 1**）/ 异常 0——判定正确。
- validate（旧）：**可写=0 / 幂等跳过=0 / 冲突=4**——4 笔可填**全被挡成冲突**。
  报文：`是否结账 表里='否' 计划='是'；计提 表里='' 计划='836.64'；…`

## 二、根因（扒 `validate_plan.py` 确认，非猜）

旧 `check_one` 判"这行填没填过"：`filled = 五列(计提/回款明细/是否结账/收款时间/收款方式)里任一非空`。
而她盈亏「明细」里**「是否结账（是/否）」是预置默认值**——单子交付了、钱还没到的行**默认就是「否」**。

**全表 5002 行实测统计**：
| 是否结账 | 回款明细 | 行数 | 含义 |
|---|---|---|---|
| 是 | 有值 | 2917 | 已收、已填 |
| **否** | **空** | **1973** | **还没收款的挂账行（预置默认）** |
| None | 空 | 107 | 空白 |

→ 每一个"还没填"的行都带着「否」，旧规则把它当"已填过"，`if not filled` 永远为假 → 永不"可写"，
只能落"跳过"或"冲突"。**金标 7-22 照不出**：那批她早填成「是」+回款有值 → 五列全对得上 → 判"跳过"，
没事。真实新数据（「否」+回款空）才暴露。

## 三、修法（不动「否」语义）

`check_one` ③：判"填没填过"的证据列**剔除「是否结账」**，只看**回款证据列**（计提/回款明细/收款时间/收款方式）：

```python
evidence_cols = [k for k in FIVE if k != "是否结账"]
filled = [k for k in evidence_cols if _norm(row.get(k)) not in ("", "None")]
if not filled:
    return {"verdict": "write", "reason": "回款列为空，可写（是否结账为她表预置默认，不计入已填证据）"}
```

下面的 same/diff 仍比对全五列（含是否结账）——**她填了钱却没翻「是」这种真不一致，照样报冲突**。

**关键：不是无脑把「否→是」**。是/否 由 classify 按"收没收满"算：
- 整单收满 → 计划 是否结账=「是」、计提填 → validate 放行写入（否→是）
- **没收满（部分回款）→ 计划 是否结账 保持「否」、计提留空、只填回款** → validate 放行写入（否 原样保留）
- 没有回款匹配的「否」未收行 → 根本不在计划里，程序不碰

→ 明妹描述的"应收100回60：原行改60结账「是」/新开40行结账「否」"那套拆行（E5）**完全不受影响**，
仍是挂起、由她手动拆，程序只填整单收满的行。
