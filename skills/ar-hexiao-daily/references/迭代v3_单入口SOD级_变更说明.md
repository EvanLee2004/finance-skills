> 触发：2026-07-23 明昊在明妹工位实测 v1.0.18，当场暴露多个问题；
> 事后拿她给的真实副本 + 智云接口实调定位，发现**取数架构本身是错的**，不是打补丁能修的。

## 二、取数换架构

**入口只有一个**：财务管理 →「回款记录」，**只筛核销日期 = T-1 一个条件**。
然后把每笔点进去的明细页抓下来——这正是明妹手工在做的动作：

```
回款记录（1 笔到账）
  ├─ 关联「下单」                    → SO + 每个 SO 的交付额
  ├─ 关联「订单同币种核销明细信息」  → 逐 SO 本次核销额；**0 行 = 全额核销**
  └─ 由 SO 查「订单明细」表          → SOD + 逐 SOD 交付额  ← 新增，SOD 的唯一来源
```

**废弃**「回款核销对账」表：对预存/预收 0 行、拿不到 SOD、且是独立筛的第二张表可能不同步。

## 三、判定下沉到 SOD

```
ΣH ≤ 到账 + 手续费                       ← 她的铁律
每笔到账至少产出 1 条判定，否则 CoverageError 报错退出   ← 防 #1 静默丢单
每个 SO：H[so] == Σ该SO的SOD交付额 → 全部 SOD 入选；否则凑唯一子集；凑不出才挂起
盈亏定位：① SO+应收金额 ② SOD ③ SO ④ 整段逐位对齐
```

**整段逐位对齐**（消歧，不是猜）：她这个 SO 的全部行（行号升序） vs 智云全部 SOD（编号降序），
两边都源自月初同一份「交付数据」，天然同序。两种判据：

- `exact`：金额序列逐位相等（SO26040297 12 行、SO26040481 10 行实测吻合）
- `ratio`：有几位不等但**所有不等位比值完全一致**＝系统性口径差，不是行错位
  （SO26040322：智云 477.61/661.31 vs 她表 488.64/676.58，两处比值都是 0.977433；
  她当天就是按智云金额填的。行错位凑不出同一个比值，所以可证）

## 四、其它

- `pay_way` 删规则甲，只留「到账月 ≠ 核销月 → 冲预收」
- `build_worklist` 消费 `validate_plan` 结论；产出合并成**一份《核销日清》**（第一列单号、第二列状态）
- `flow_ledger` 三键改用**到账总额**
- `verify_sources` 忽略 `备份/`；`apply --in-place` 成功后**重打指纹**（否则下一次 verify 必报假警报）
- 删除 `build_review_sheet.py`（并入 build_worklist）
