# 社保公积金合并与在职台账

财务部每月把各主体（甲骨易/上海/湖南分公司/湖南子公司等）的社保台账、公积金台账"单位缴纳部分"金额，合并进当月工资表，并维护一张跨月累计的薪酬明细+组织架构名册，自动算每个人的在职月份。

输入是一张干净工资表（清洗过、含"姓名"+"基本工资"表头即可），这个技能把社保/公积金"单位缴纳部分"合并进去、再叠加跨月累计+在职判定逻辑。

## ⚠ 开工前必读：这是"先做出来试"的版本，不是走完需求确认书流程的成熟技能

2026-07-02 按"照录音+明妹操作录屏演示做个skill出来"直接开发，**没有先出需求确认书让明妹/亮晶书面确认过**。这意味着：
- 里面几条关键业务规则（尤其是"在职状态怎么判定"，见 `config/业务规则.md` 第三节）是从**一次录屏演示反推**的，没有书面拍板过。
- **每次用完都要主动跟明妹核对结果对不对**，跟她说清楚这是新技能第一次真刀真枪跑、需要她把关，不是"确认过的东西照做"。
- 发现结果跟她实际口径对不上，别自己悄悄改代码将就，去改 `config/业务规则.md` 记录下来、跟她核对后再定。

## 你（agent）该怎么干

**核心原则：用它的是财务同事，不是技术员。** 她给到工资表+社保/公积金台账，你把活办完；别让她自己找列号、认sheet名。

### 1. 认清输入（按内容认，不靠文件名）
需要三样：
- **该主体当月工资表**：可以是清洗过的干净工资底表（`主表` sheet），也可以是任何含"姓名"+"基本工资"表头的工资表。
- **社保/公积金台账**：分两种拿法——① 独立两个文件（多数主体：甲骨易/上海/文化传媒/园创园/语丰泰达），传 `--insurance` + `--fund`；② 合并成一个文件（湖南分公司/湖南子公司这类"五险一金台账"），传 `--combined`。
- **主体名**（`--entity`）：中文全称，如"甲骨易""上海""湖南分公司"。

```bash
python3 "<本skill目录>/scripts/merge_insurance.py" --inspect --input-dir <放文件的绝对目录>
```
`--inspect` 会列出每个文件识别到几个 sheet、每个 sheet 是否含"单位社保小计列"/"单位公积金列"、推测的月份。**新主体第一次用（没在 `config/主体配置.json` 的"已配置主体"里）务必先跑这个看识别对不对，别直接信正式跑。**

### 2. 月份必须显式传，别让脚本自己猜
```bash
python3 "<本skill目录>/scripts/merge_insurance.py" \
  --entity 甲骨易 --month 202606 \
  --payroll <该主体当月工资表.xlsx> [--payroll-sheet <sheet名>] \
  --insurance <社保台账.xlsx> --fund <公积金台账.xlsx> \
  [--master <累计薪酬台账.xlsx>] [--out <输出.xlsx>]
```
湖南这类合并台账用 `--combined` 代替 `--insurance`/`--fund`：
```bash
python3 "<本skill目录>/scripts/merge_insurance.py" \
  --entity 湖南分公司 --month 202606 \
  --payroll <工资表.xlsx> --combined <五险一金台账.xlsx> \
  [--master <累计薪酬台账.xlsx>] [--out <输出.xlsx>]
```
- 工资表如果有多个"长得像工资表"的 sheet（比如把好几个主体堆在一个大杂烩工作簿里），且月份/主体锁不定，脚本**不会瞎猜**，会报错列出候选、要求用 `--payroll-sheet` 明确指定——把候选转述给她，让她说清楚指哪个。
- `--master` 是跨月累计的薪酬台账（脚本自己上次的输出就行）。**不传就只出这一个月的单月结果**；传了会自动把这个月并进去、组织架构增量维护、在职月份重新算。**同一个月+主体重复跑是幂等的**（会先把旧的这个月记录删掉再重新写，不会累加出两份）。

### 3. 输出怎么看（6个sheet）
- **薪酬明细**：跨月累计明细（每人每月一行），核心产出。
- **组织架构新**：只增不删的姓名→地区/部门/岗位名册。
- **编外人员**：`config/编外人员名单.json` 里标记的人，单独列出，不进薪酬明细。
- **待人工核实**：匹配不到单位社保/公积金的人 + 社保公积金台账里有但工资表里没有的人，都在这，交人工判断（新员工没入台账？编外没在名单里？打错名字？）。
- **薪酬汇总**：多月横向对比的宽表（跟她自己手工做的历史真实产物列结构一致），只保证数据对，**没有仿她手工配的橙绿隔月颜色**（这条不是核心需求，优先级低）。
- **运行报告**：列识别用了哪一列、诊断信息。

### 4. 出活后怎么收尾
1. **一句话人数分布**：这月XX人处理完了，待人工核实XX条（说清楚是"没配到社保"还是"台账有人工资表没有"）。
2. **⚠ 特别提醒她核对"在职状态"这一列**：这是从她操作录屏反推的规则（发工资=在职），第一次用**务必让她逐条看几个真实案例对不对**，尤其是有过断档/入离职当月的人。
3. **新主体第一次用**，无论 `--inspect` 看着识别对不对，第一次正式跑完都建议她抽查几个人的单位社保/公积金金额是否和原台账一致。

### 5. 口头改规则 → 改 config（活表，别碰代码）
她说"这个人不是编外了""在职状态不该这么判"——去改 `config/编外人员名单.json` 或 `config/业务规则.md` → 重跑 → 给她看 → `git add config/ && git commit` 固化。

## 会变的东西在哪（改表不改码）
- **编外人员名单** → `config/编外人员名单.json`
- **在职判定口径 / 已验证过的主体清单 / 已知不做的范围** → `config/业务规则.md`
- **主体格式说明**（哪些验证过、哪些没有） → `config/主体配置.json`
- 确定性算法（表头识别、单位金额列关键字匹配、跨月累计、幂等去重）写死在 `scripts/merge_insurance.py`

## 验收口径（判断脚本对不对）
**已知基准**（2026-07-02 真实数据核实，链路：湖南分公司干净工资底表 + `湖南分公司五险一金台账202606.xlsx`）：
- 82人全部处理，薪酬明细行数=82（单月）。
- **谢柯**：单位社保小计=1133.01、单位住房公积金=105（与源台账 `2026-6` sheet 逐格核对一致）。
- 跨月累计+幂等重跑：连续跑 202605→202606→重跑202606，薪酬明细总行数稳定在164（82×2月），不会因重跑翻倍；组织架构名册稳定在82人。
- 跑 `tests/test_robustness.py` 应全绿（4种真实表头结构的合成金标 + 真实数据冒烟）。

> ⚠ 数据敏感：含身份证号/工资/社保金额 PII，只在本地跑、结果别外传。
