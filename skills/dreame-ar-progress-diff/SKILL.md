# 追觅应收进度对比

> 读完本节应能**冷启动干活**：不用再让亮晶从零解释「表长啥样、要比啥、怎么算变化」。  
> 细节口径与坑见 `references/SOP_追觅应收进度对比.md`（异常时再打开）。会变参数在 `config/`。

---

## 0. 任务一句话（你在干什么）

亮晶跟进**追觅客户**各业务线的回款/PO/发票进度。她会不定期导出一份 list 工作簿；其中一张 **「应收进度」** 是总览宽表。  

**本活 = 把两份（或更多）不同时间导出的「应收进度」并排 diff**，回答：

- 哪些格子的**文字/金额**变了？  
- 哪些格子的**底色（状态）**变了？  
- **列**有没有增删、改名（对接人名字变了也要对上）？  
- 有没有**新账期行**（后期表多出来的月份）？  

产出：一份 Excel 对比报告 + 给人看的短结论。  
**不是**做公司通用应收合并（那是 `receivables-merge`），**不是**改源表、不是核销银行流水。

---

## 1. 表长什么样（稳定结构 · 别死记某次文件名）

| 维度 | 你怎么认 |
|------|----------|
| 文件 | 通常是追觅 list 工作簿；**别靠文件名硬编码**。按内容：有「应收进度」类 sheet 即可。同一批任务里 ≥2 个时间点快照。 |
| 目标 sheet | 默认名「应收进度」（可在 config 改）；找不到再模糊匹配含「应收进度」的，或按列特征打分。 |
| 行 | **A 列 = 账期**（如「去年10月」「1月」「6月」…）。读到图例/汇总停止词为止（config 里可加）。 |
| 列 | **从 B 列起 = 业务线/对接人**。一列「金额」，右侧常跟 0～3 个子列：PO时间、发票上传时间、预计付款时间。末尾也可能有**只有金额、无子列**的窄列。 |
| 底色 | 单元格填充色表示状态（绿/黄/橙/灰等）。中文名映射在 `config/颜色图例.json`，**以表为准、以 config 为准**，别臆造新语义。 |
| 公式 | 「预计付款」里常见 `=某格+60`。业务上这是自动推算日期，**不算人工备注变更**（config：忽略公式=是）。 |

**会变、所以不要写死进脑子/代码：**

- 有几个人名列、叫什么、谁改名、谁列被删  
- 每次导出有几个月、有没有新月  
- 具体哪笔回款、催促文案、金额数字  
- 文件叫 Martin0709 还是别的  

这些都从**当次文件解析结果**出，不要预设「永远 12 列 / 永远三份 / 永远某个人」。

## 2. 用户要的结果（Done 长什么样）

1. **Excel 报告**（脚本出，你别手搓对比逻辑）大致含：  
   结论摘要 · 列结构对比 · **值变化** · **颜色变化** · 预计付款时间 · 明细对比 · 运行报告  
2. **人话结论**（你写）：数字摘要 + 几条业务向要点（催促 / 回款核销 / 收 PO / 新账期 / 列变动）。  
3. 源 xlsx **只读不改**。产物路径明确告诉她。  

她主要看 **「值变化」「颜色变化」**；明细备查。

## 3. 你怎么干（固定流程）

**原则：脏活归脚本，判断/沟通归你。** 别让她改文件名、看命令、数 sheet。路径一律**绝对路径**；`<本skill目录>` = 本 SKILL.md 所在文件夹。

### 步骤 A — 找齐 ≥2 份源表

```bash
python3 "<本skill目录>/scripts/compare.py" --inspect --input-dir <目录绝对路径>
```

- 默认也可扫 `工作区/input/`。  
- 跳过：`~$` 锁文件、文件名含「对比报告/对比结论」的旧产物。  
- 按内容认「像进度表」的 xlsx；**按时间从旧到新**排序（文件名里的月日、或她指定的顺序）。  
- 够清楚 → 一句话告知「我按 A→B→C 比」就开跑；不足 2 份 / 分不清谁旧谁新 → **才问她一句**。

### 步骤 B — 跑对比（禁止另写一套 diff）

```bash
# 最稳：旧 → 新 显式点名
python3 "<本skill目录>/scripts/compare.py" \
  --files <旧.xlsx> <中.xlsx> <新.xlsx> \
  --out <交付目录>/追觅应收进度对比报告.xlsx

# 或目录自动认 + 文件名日期排序
python3 "<本skill目录>/scripts/compare.py" --input-dir <目录> --out <路径.xlsx>
```

- 支持 **2～N** 版，不强制 3。  
- `--out` 可省：默认落在最新源文件同目录（带时间戳）。  
- **stdout = 产出路径**；统计在 stderr / 报告「结论摘要」「运行报告」。

### 步骤 C — 用报告做结论（你读 sheet，别重算格子）

从报告里提炼，**按当次数据说话**：

| 看哪里 | 提炼什么 |
|--------|----------|
| 结论摘要 / 统计 | 值变化几处、底色几处、付款备注几处、公式忽略几处 |
| 列结构对比 | 改名、某版缺失、仅后期才有的列 |
| 值变化 | 文案关键词模式：催促、回款/核销、收到 po/PO、发送对账单、金额数字变动、新期间有金额 |
| 颜色变化 | 结合 config 图例：待付/未收PO/流程中/已回款 等状态迁移 |
| 运行报告 | 对齐存疑、公式忽略清单 → 有则问人，无则带过 |

**结论模板（填当次数，别抄历史案例）：**

> N 版（标签…）对比完：值变化 X、底色 Y、预计付款备注 Z（公式忽略 W）。  
> 要点：① … ② … ③ …（3～6 条业务向，优先回款闭环 / 收 PO / 新账期 / 密集催促 / 列结构）  
> 报告：`<绝对路径>`。建议先看「值变化」「颜色变化」。

可选：她要 Word 再用 docx 技能整理同一套结论（**内容跟当次报告走**，别贴旧样本数字）。

## 4. 脚本已钉死的口径（你信它，别另算）

这些是**稳定性核心**，写在 `scripts/compare.py`，不靠你临场发挥：

1. **期间 = 各版本并集**（后期多出来的月份必须进对比，禁止只拿第一份的期间）。  
2. **人名列一对一智能对齐**（去空格/换行/全半角括号；括号里加人名仍算同一业务列）。对齐存疑进运行报告。  
3. **预计付款：公式当空**，只比人工备注文本。  
4. **值变化**：规范化后的非空文本/数字/日期不一致。  
5. **底色变化**：在「两边都有该期间」的版本之间比 RGB。  
6. **源文件只读**。

表头别名、停止词、颜色中文名、是否忽略公式 → **`config/`**，改表不改码。

## 5. 什么时候问人 · 什么时候自己扛

| 自己扛 | 问她（一句、可秒回） |
|--------|----------------------|
| 认 sheet、解析列、跑脚本、写统计与模式摘要 | 不足 2 个文件 / 分不清时间顺序 |
| 文件名日期能排序 | 运行报告有**对齐存疑**（两个候选同分） |
| 按关键词归类变化（催促/回款/收PO/新月） | 某列整列消失或清空，业务含义她才知道 |
| 颜色按 config 图例称呼 | 她说表头/状态色改了 → 你改 config 再跑 |

**不要问：** 命令怎么写、sheet 叫什么、要不要比底色（默认要比）、公式算不算备注（默认不算）。

## 6. 像同事一样收尾（主动但克制）

1. **有变化才展开**：没异常就「X 处值变化 / Y 处底色，结构稳定，报告在…」。  
2. **模式优先于流水账**：先回款闭环、收 PO 推进、新账期、列增删；再提零散备注。  
3. **金额细微尾差**单独点一句请她确认，别淹没在催促列表里。  
4. **本岗位留心眼**：  
   - 文案含回款/核销 + 色变浅/灰 → 提示可归档闭环  
   - 未收 PO 色 → 流程中色 + 文案收 PO → 提示跟付款节点  
   - 大面积「催促」→ 这是跟进周，未必是底表重做  
   - 新期间有金额 → 提醒纳入当期跟踪

## 7. 会变 vs 固定（别写死会变的）

| 固定（脚本/本 skill 流程） | 会变（config 或当次数据） |
|----------------------------|---------------------------|
| 并集期间、对齐算法、公式忽略、报告结构、跑法 | 列名/人数、期间列表、具体金额与备注 |
| 「要出值+色+列结构」 | 颜色 RGB→中文（`颜色图例.json`） |
| 源只读、≥2 文件 | 子列表头别名（`子列识别.json`） |
| | sheet 名、表头行、停止词（`业务规则.md`） |

她口头说「表头改名了 / 多了种颜色 / 停止词要加」→ 改对应 config → 重跑 → 确认后再 commit config（若在 git 库内）。
