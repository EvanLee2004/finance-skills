# 工资表清洗与信息匹配

财务部每月处理湖南分/子公司工资表的活自动化：清洗明妹给的原始表格式，按姓名从权威花名册匹配电话/部门/岗位/地区，出一张干净底表。

## 你（agent）该怎么干

**核心原则：用它的是财务同事，不是技术员。** 她扔两个 Excel 进来，你把活办完；别让她改文件名、看列号、看 sheet 名。

### 1. 找齐两个输入（按内容认，不靠文件名）
需要：①**工资原始表**（明妹给的未处理版：多 sheet，一 sheet 一公司/一月，含姓名/身份证号码/基本工资~实发工资）②**员工信息表（权威）**（sheet 名全是6位数字 YYYYMM，含姓名/地区/部门/岗位/身份证号/电话）。

⚠️ **路径全用绝对路径，别 cd。** 脚本按自身位置自动找 `config/`。下面 `<本skill目录>` 换成真实绝对路径。

```bash
python3 "<本skill目录>/scripts/clean_match.py" --inspect --input-dir <放文件的绝对目录>
```
- 默认找 `工作区/input/`；文件在别处就把 `--input-dir` 指过去。
- `--inspect` 会列出：识别到的工资 sheet + 猜的所属公司、月份线索（sheet名数字 + 表头区疑似日期序列号交叉验证后的建议月份）。
- **两个输入候选都恰好1个才算认清楚**；出现2个以上同类候选（比如目录里混了别的月份的工资文件），脚本不猜、报出来交人确认，你转述给她，让她说清楚指哪个。

### 2. ⭐ 月份必须让人确认一句，别自己替她定（硬性要求）
`--inspect` 给出的"建议 --month"只是线索交叉验证后的推测，**不能直接当真去跑**——月份判断错了，会悄悄把人匹配到错误月份花名册、填错部门/电话，且**没有任何报错信号**，比"匹配不上"更隐蔽危险。
- 跑完 `--inspect`，把建议月份**说给她听、等一句确认**（例如："看着是处理6月的，我按 202606 跑，对吗？"），她确认或改口后，才带着确定的 `--month` 去跑第3步。
- 正式跑（非 `--inspect`）不传 `--month` 会直接报错退出，这是设计如此，不是 bug。

### 3. 跑清洗匹配
```bash
python3 "<本skill目录>/scripts/clean_match.py" \
  --payroll <工资原始表路径> --employee <员工信息表路径> --month <YYYYMM> \
  --out <交付目录>/工资底表_<YYYYMM>.xlsx
```
- `--payroll/--employee` 省略则按内容从 `--input-dir` 自动认（仅当候选唯一时）。
- `--out` 可省：省了落在工资原始表同目录。
- 脚本自动：认工资 sheet（按列特征：姓名+身份证号+≥2个金额列，不靠sheet名/顺序）→ 定人员行范围（遇『合计』行或连续多行全空即停，参考数据块不会被读进来）→ 检测无表头空列（100%空+左右表头都非空才确认，多候选不猜）→ 公式转数值两位小数（缓存值缺失标"数值缺失-待核实"，不当0）→ 合计行反算校验(容差0.05) → 按姓名匹配当月花名册 → 出4个 sheet：**主表 / 待人工核实 / 匹配成功 / 运行报告**（脚本跑完的最后一行日志会原样报这4个名字——跟同事汇报时照抄这4个 sheet 名，别自己改写/翻译成别的说法）。

### 4. 匹配逻辑摘要（脚本自动做，参数在 `config/业务规则.md`）
- 匹配键 = **姓名**（花名册内部若有同名，先按身份证号去重——如果这几条记录身份证号完全一样，说明是同一人被花名册重复登记，自动算匹配成功；身份证号不同才是真重名，才标『重名-待核实』、不自动填、列出候选人的部门/岗位/证号尾4位交人工判断）。
- 花名册**只用处理月份对应的那一个 sheet**，不跨月扫描——因为同一人每月都会在花名册里出现一次，跨月扫会把"这人在好几个月份sheet里都有"误判成"重名"。
- 花名册当月查不到姓名 → "未匹配-待核实"，不瞎填。真实数据实测(2026-07-01)：82人里9人查不到，属正常现象(新员工没来得及录入花名册)，不是技能故障。

### 5. 出活后怎么收尾（人话、给摘要、点出真正该关注的）
跑完别只吐"完成了"，说清楚这几件事：
1. **一句话总人数分布**："这月 XX 人处理完了：匹配成功 A、未匹配 B（这些多半是新员工，建议她去问HR要资料）、重名 C（列出来了，麻烦她一句话定哪个）。"
2. **合计校验结果**：主动说"两个sheet(公司)的合计我都反算比对过，都在容差内/其中XX有差异"——这是抓"数算出来的可离谱"的关键一步，别漏说。
3. **⭐ 数值缺失别一带而过、但也别自己下结论**：运行报告里『数值缺失-XX列』的统计，如果集中在**养老/失业/基本医疗/住房公积金**且都是实习生 → 大概率正常（实习生不缴社保），提一句就行；但如果**个人所得税/税前工资/实发工资**这类核心列也大量缺失（2026-07-01真实数据踩过：77人里44人个税缺失）→ 这通常是原始表的公式没被重新计算过（明妹粘贴后没在Excel里刷新过），**主动建议她回原表触发一次重算再重新导出**，而不是我们这边猜一个值填上。这个判断留给她，别自己悄悄拿0或猜测值顶上。
4. **待人工核实sheet的范围提醒**：这个sheet不只是"没匹配上的"，还包含"匹配成功但某些数值缺失"的行——跟她说清楚这个sheet两种情况都有，别让她误以为都是没匹配上。

### 6. 口头改规则 → 改 config（活表，别碰代码）
她说"合计容差放宽到1块""这个子公司sheet名不叫湖南科技了改叫XX""要不要保留原表的错误部门做审计对比"——去改 `config/业务规则.md` 或 `config/列名别名.json` → 重跑 → 给她看 → `git add config/ && git commit` 固化。**绝不把规则写死进脚本。**

## 会变的东西在哪（改表不改码）
- **合计容差/小数位/合计行关键字/空行停止阈值** → `config/业务规则.md`
- **两张表的列名别名 + 所属公司映射（sheet名前缀→公司全称，扩展到10个主体时在这加行）** → `config/列名别名.json`
- 确定性算法（sheet识别、人员行边界、无表头空列检测、公式转数值、匹配三分类决策树）写死在 `scripts/clean_match.py` 保证可复现。

## 验收口径（判断脚本对不对）
**已知基准**（测试数据 `明妹工资原始表_未处理版.xlsx` + `湖南分公司员工信息（财务）.xlsx`，2026-07-01 人工核实）：
- 工资表识别到 2 个 sheet（`6月湖南` 77人、`6月湖南科技` 5人），共 **82 人**；两个 sheet 的合计行反算校验都通过（容差0.05内）。
- 月份 = 202606（花名册按此月 sheet 匹配）：**匹配成功73、未匹配-待核实9、重名-待核实0**。
- 关键案例：**谢柯**→匹配成功，部门=人力中心/岗位=HRBP/地区=湖南（电话字段也应正确填入，具体号码属PII不在此列出）；**罗欢**（6月湖南科技，花名册202606未收录）→未匹配-待核实。
- 跑 `tests/test_robustness.py` 应全绿（合成金标 + 真实冒烟）。

换别月数据跑：看识别 sheet 数无告警、合计校验通过、关键案例对、"未匹配"都是真没在花名册里的人，即结构稳。

> ⚠ 数据敏感：工资表含身份证号/工资金额 PII，**只在本地跑、结果别外传**。
