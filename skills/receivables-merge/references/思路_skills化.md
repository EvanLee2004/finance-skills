# 思路 · 把应收账款合并做成 skill

> 配套 `SOP_应收账款合并.md`（流程依据）与流程图。目标：让 agent 用 python 按 SOP 完成任务。

## 一、形态：一个自包含的 skill 文件夹

```
skills/receivables-merge/
├── SKILL.md              ← agent 行为指南：何时触发、怎么对话、何时停下问人
├── SOP_应收账款合并.md     ← 流程依据（本目录已有）
├── scripts/
│   └── merge.py          ← 按 SOP 实现 S1–S8 的 python（确定性逻辑）
├── config/
│   ├── 营销人员应收匹配规则.xlsx   ← 维护表（S4 归属，已拷入）
│   └── 列名别名.yaml              ← S1 认列的别名表
└── 工作区/
    ├── input/   源台账.xlsx · 回填源.xlsx
    └── output/  应收all.xlsx（17列主表 + 透视Sheet2）+ 复核报告
```

## 二、分工：python 干确定的，agent 干会变和把关的

- **python（确定性，照 SOP 跑）**：S1 合并 / S2 账龄 / S3 回填 / S4 归属（读维护表）/ S5 删0行 / S7 排序 / S8 透视。这些口径都已被视频+成品逐行验证，机器比人准、比人稳。
- **agent（编排 + 异常 + 把关）**：
  - 听懂"跑本周应收" → 检查三个输入齐不齐 → 调 merge.py；
  - 碰到**别名表里没有的新列名**、或**归属拿不准** → **停下来问人，不瞎来**；
  - 把**复核报告**（#N/A 清单、年度告警、被删0行清单）呈给人过目。

## 三、关键设计：把"会变"全外置成「你能维护的表」

整个流程只有两处会变，都做成**配置表**，规则变了改表、不改代码：
1. **列名别名表**（S1 认列）：把"应收金额/订单折合本币/金额"、"交付月份/项目交付/销售确认"这类归一规则写进 `列名别名.yaml`。
2. **维护表**（S4 归属）：`营销人员应收匹配规则.xlsx`，你维护"离职→接手""客户→重分配"。

这样新表换了列名、销售有了变动，**你改表就行**，不用动 agent / 代码。

## 四、安全红线（与平台一致）

- 源数据**只读**；产物只写 `output/`，带时间戳防覆盖。
- **删 0 行要留底**：被删的应收=0 行单独存一份清单，可追溯（不是真删没了）。
- 维护表/真实数据**不进 git**。
- **对外发送前人工过目**复核报告（关键动作人决定）。

## 五、验收标准（达到才算"好 skill"）

用 `原始素材/2026.6.4日应收.xlsx` + 回填源 + 维护表跑一遍，产物**逐项复现** `2026.6.4应收all.xlsx`：
- 主表 **3243 行**、17 列、应收=0 行 **= 0**；
- 高美杰三路分布：`梁玲玲-高美杰`104 / `于占国-高美杰`74 / `高美杰1`10；
- 无离职销售光名残留；
- Sheet2 透视：应收金额 按 销售人员→客户。
**能 100% 复现才落地，复现不了不上线。**

## 六、与旧脚本的关系

旧 `merge_receivables.py` 已覆盖 S1–S3 + 名称变体检测，但**缺 S4 归属 / S5 删0行 / S8 透视**，且 S4 旧版只"检测变体不归属"。新 skill = 在其基础上补这三步并接入维护表，或按本 SOP 重写一版干净的 `merge.py`。
