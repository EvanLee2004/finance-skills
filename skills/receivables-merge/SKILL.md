# 应收账款合并（赵 Excel 流程）

把财务部每周手工做的"应收账款合并"自动化。完整流程与口径依据见 `references/SOP_应收账款合并.md`（**动手前先读它**），设计思路见 `references/思路_skills化.md`。

## 你（agent）该怎么干

**核心原则：人说人话，找文件/认文件/跑脚本/出复核 全归你，别让用户改文件名、摆文件夹。**

### 1. 找齐三个输入（按内容认，不靠文件名）
需要：①**源台账**（本周原始，多个年份 sheet）②**回填源**（上一版做好的 all，带销售标注列；**还用于结转老坏账**）③**维护表**（`config/销售归属维护表.md`，两段：离职→接手 / 客户→接手；默认已在 `config/`）。

⚠️ **先 `cd` 到 skill 根目录再跑所有命令**（命令里有相对路径；不 cd 会找不到脚本/配置）。脚本自带的 `config/` 不受工作目录影响，但**别再手动传 `--rules`**——省略它就自动用 skill 自带的维护表，最稳。

```bash
cd "/Users/evanlee/Documents/甲骨易实习/2_战略专项/财务部Agent员工/skills/receivables-merge"
# 先识别看认得对不对（源台账/回填源按内容认）
python3 scripts/merge.py --inspect --input-dir <放文件的绝对目录>
```
- 默认去 `工作区/input/` 找；用户文件在别处就把 `--input-dir` 指过去（**用绝对路径**）。
- 认出来后**跟用户核一句**"源台账=X、回填源=Y，对吗"。**认不准 / 缺文件 / 有歧义 → 问用户，不要猜。**

### 2. 跑合并（用绝对路径、别传 --rules）
```bash
python3 scripts/merge.py \
  --source <源台账绝对路径> --ref <回填源绝对路径> \
  --out 工作区/output/应收all_<日期>.xlsx
# 账龄基准月默认从源文件名取；补往月用 --base-month 202606
```
- **不要传 `--rules`** → 自动用 `config/销售归属维护表.md`（绝对解析、最稳）。万一你确实要指定，必须用**绝对路径**，否则相对路径在错误工作目录下会报错。
- `--ref`（上一版 all）省略则跳过回填**和老坏账结转**（标注列留空、缺老年份账）。**要完整复现成品，必须给 `--ref`。**
- 脚本按 SOP 跑 S1–S8：合并 → 账龄 → 回填 → **销售归属（维护表驱动）** → **结转老坏账** → **删应收=0行** → 复核 → 排序 → **透视汇总**。

### 3. ⭐ 确认销售归属（每次必做，不许跳过）
**销售变更是「会变、要人确认」的事，绝不能由脚本/你私自拍板。** 跑完后：
1. 打开产物里的 `归属变更`（这次准备把谁→归给谁、各多少行）和 `归属存疑`（疑似错别字/离职名对不齐，如「牛雪如 vs 牛雪茹」）两个 sheet。
2. **明确摆给用户**："这次按销售维护表，准备这样归属（列出变更）；另有这些存疑（列出）。**确认还是要改？**"
3. **用户说"确认"** → 就照这个定稿，往下走。
4. **用户说要改**（如"骆利飞归王雄""牛雪如就是牛雪茹"）→ 按用户说的**改 `config/销售归属维护表.md`**（加人/改名/调接手）→ **重跑** → 再确认。
5. **改完把改动固化**：`git add config/销售归属维护表.md && git commit`，让这次的销售变更留痕、下次自动生效。

> 维护表是"活"的：每周可能有人离职/客户转手。**你的职责是把变更和存疑摆清楚、让人一句话拍板，然后忠实落到表里**——不是自己猜规则、不是写死在代码里。

### 4. ⭐ 先看「认列告警」，再交付其余复核
**跑完第一眼看产物里的 `认列告警` sheet（运行日志也会打 ⚠）**——这是稳定性的关键闸：
- **「关键列没认出」**：源表某个 sheet 改了列名（如"应收金额"→"应收款"），这批数据会**丢失/算错**。→ 把新列名加进 `config/列名别名.json`，重跑。
- **「未知 sheet」**：源表多了个不是年份、也没登记的 sheet（如"9月批量"）。→ **别猜**，问用户是不是新特殊批次、年度记啥/整批交付月哪个月，确认后补进 `config/业务规则.md`，重跑。

其余复核：`未匹配复核(#N/A)`、`离职残留`（应为空）、`被删0行`（留底）、`年度校验`、`透视汇总`、`运行报告`。**主动提醒用户重点看**认列告警、未匹配、被删0行。

## 会变的东西在哪（都是活的，改表不改码）
- **列名别名**（认列规则）→ `config/列名别名.json`
- **销售归属**（离职→接手、客户→重分配）→ `config/销售归属维护表.md`（用户维护、每次确认）
- **特殊批次 / 跳过的 sheet / 业务规则说明** → `config/业务规则.md`（agent 跑前先读；遇表里没覆盖的情况停下问人）
确定性算术（账龄、合并、删0、结转、求和）写死在脚本里保证可复现；**会变/要判断的都在上面这几张 MD/表里**。

## 验收口径（判断脚本对不对）
**已知基准**（源台账 `2026.6.4日应收` + 回填源 `2026.5.28应收all`，`--base-month 202606`）：
- 主表 **≈3247 行**；高美杰三路 **梁玲玲-高美杰 104 / 于占国-高美杰 74 / 高美杰1 10**；删0行 **175**；结转老坏账 **29**；离职残留 **0**。
- ⚠ **3247 vs 成品 3243 的 +4 行是正常的、已知的**：成品的 3243 行**一行不缺都在**（成品有·我无=0），多出的 4 行是 ¥0.61/¥1 这类极小额（成品删了、脚本保留）+ 结转边界，不是错。**别因为对不上 3243 就以为跑错了。**
- 还剩 6 行销售归属与成品不同（特变/广州库洛）= 成品自身不一致，等亮晶姐定，见 `20260618_应收归属待确认_问亮晶姐.docx`。

换别的周数据跑：看 `认列告警`（应为空）、`离职残留`（应为空）、年度分布合理、不崩，即结构稳。
